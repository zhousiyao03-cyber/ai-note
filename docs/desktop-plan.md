# Plaud Desktop 项目规划

## 目标

在现有 web 版基础上，规划一个可独立交付的 `plaud desktop` 客户端，优先满足以下目标：

- 复用现有账号体系、文件、转写、标签、Ask AI 和设置能力
- 提供比浏览器更强的桌面体验，例如本地文件接入、后台上传、系统通知和快捷操作
- 保持后端统一，避免再维护一套独立服务
- 为后续 Windows 优先、再扩展 macOS 留出空间

## 当前仓库现状

基于现有代码，仓库已经具备 desktop 项目可复用的核心基础：

- `app/` 是现有主应用，技术栈为 Next.js 16 + React 19 + TypeScript
- 前端已经有文件列表、详情页、转写编辑、Ask AI、设置、回收站等模块
- API 路由和 Supabase 集成已在仓库内推进，适合继续作为统一后端
- `docs/` 已有 backend architecture、frontend integration 和 phase progress 文档

这意味着 desktop 不应该从零做一套产品，而应该做成“新客户端 + 复用现有业务能力”。

## 技术路线建议

### 推荐方案

采用 `Tauri 2 + React + Vite + TypeScript` 新建桌面端工程，和现有 `app/` 并行存在。

推荐原因：

- 安装包和内存占用通常比 Electron 更轻
- 原生能力足够覆盖文件系统、通知、自动更新、窗口管理等需求
- 可以把桌面能力显式收敛在 Rust/Tauri command 层，不污染 web 代码
- 当前 web 端已经有完整业务模块，适合抽共享层，而不是直接把 Next.js 硬塞进桌面壳

### 不推荐作为主路线的方案

直接用 Electron 包现有 Next.js 页面，虽然启动快，但中期成本偏高：

- 包体积更大
- 运行资源占用更高
- 容易把桌面逻辑和 web 逻辑混在一起
- 后续做安全隔离、自动更新和原生能力时更容易出现历史包袱

### 一句话结论

如果目标是认真做 `plaud desktop` 产品，而不是只做一个临时壳，建议走 `Tauri 2`。

## 建议的仓库结构

建议把仓库逐步整理成下面这种形态：

```text
app/                  # 现有 web + API + backend integration
desktop/              # 新的 Tauri desktop 客户端
packages/
  shared-api/         # API client、DTO、query keys、错误模型
  shared-types/       # AudioFile、Transcription、Tag、User 等类型
  shared-i18n/        # 中英文文案、语言配置
  shared-ui/          # 纯展示型组件（后续按需抽）
docs/
  desktop-plan.md
```

第一阶段不要急着抽太多共享包，避免“为了抽象而抽象”。

建议的抽取顺序：

1. 先抽 `shared-types`
2. 再抽 `shared-api`
3. 最后按实际重复度决定要不要抽 `shared-ui`

## Desktop 版本定位

桌面版不只是 web 的复制品，建议把它定位成“高频生产工具”。

### MVP 必做能力

- 登录和会话保持
- 文件列表、搜索、筛选、标签、回收站
- 音频上传
- 转写状态跟踪
- 转写详情编辑
- Ask AI
- 设置页和基础偏好项

### Desktop 独有能力

- 从本地文件系统直接导入音频
- 拖拽上传整个文件夹或多文件批量导入
- 系统通知，例如转写完成提醒
- 后台继续上传或轮询任务状态
- 打开本地最近文件
- 更稳定的长时任务体验

### 第二阶段再做的能力

- 全局快捷键打开上传窗口
- 托盘菜单
- 本地录音
- 离线草稿和失败重试队列
- 自动更新
- 启动即恢复上次工作区

## 架构原则

### 1. 后端继续复用现有服务

desktop 和 web 共用同一套：

- Supabase Auth
- Supabase Storage
- Next.js API Routes
- Inngest 转写任务

不要为了 desktop 再建一套本地服务端。

### 2. 桌面端只增加“平台适配层”

桌面端新增的内容应尽量限定在这些层：

- `platform/files`：打开文件、保存导出、选择目录
- `platform/notifications`：系统通知
- `platform/window`：窗口、托盘、快捷键
- `platform/storage`：本地安全存储、最近使用记录

业务层尽量仍然使用统一的：

- API client
- 类型定义
- query keys
- 表单校验
- i18n 文案

### 3. 鉴权不能沿用浏览器假设

桌面端要单独设计安全存储：

- access token 不要只存在普通 localStorage
- refresh token 或 session 相关信息优先放系统安全存储
- 需要明确登出时清理缓存、最近文件、临时上传记录

## 分阶段实施计划

## Phase 0: 基础决策与脚手架

目标：让 desktop 工程能独立启动，并确认共享层边界。

输出：

- 新建 `desktop/`
- 选定 Tauri 2、React、Vite、TypeScript
- 建立路由、状态管理、TanStack Query、i18n
- 接入基础 lint 和 build
- 抽出 `shared-types`
- 抽出 `shared-api` 的最小版本

完成标准：

- `desktop` 可以本地启动
- 能访问登录页和空工作区
- 可以成功调用 `/api/v1/me`

## Phase 1: 桌面 MVP 主流程

目标：跑通“登录 -> 文件列表 -> 上传 -> 转写详情”。

范围：

- 登录/登出
- 文件列表与筛选
- 文件详情页
- 上传文件
- 转写状态轮询
- 转写编辑保存
- 标签和回收站

完成标准：

- 用户可以在 desktop 完整完成一次音频上传和转写查看
- 关键页面和 web 版字段保持一致
- API 错误能正确展示

## Phase 2: 桌面增强能力

目标：让 desktop 体验明显优于浏览器。

范围：

- 文件拖拽导入
- 批量导入
- 系统通知
- 最近文件
- 上传失败重试
- 更稳的长任务状态恢复

完成标准：

- 应用重启后仍能恢复未完成任务视图
- 转写完成能触发系统通知
- 批量导入至少支持 5 到 20 个文件连续入队

## Phase 3: 高级效率能力

目标：增强日常高频使用价值。

范围：

- Ask AI 完整持久化体验
- 全局快捷键
- 托盘
- 本地录音
- 导出增强
- 自动更新

完成标准：

- 具备首个适合小范围内测分发的版本

## Phase 4: 发布与运营

目标：让桌面版具备稳定发布能力。

范围：

- 安装包构建
- Windows 签名与更新策略
- 崩溃收集和日志
- 发布通道区分（dev/beta/prod）
- 版本升级回滚预案

完成标准：

- 形成可重复的 release 流程
- 至少支持内部测试和灰度发布

## 关键模块拆分建议

### 1. 共享模块

- 类型：`AudioFile`、`Transcription`、`Tag`、`User`
- HTTP client 与错误包装
- 查询 key
- DTO 映射
- i18n 资源

### 2. 桌面端专属模块

- 文件系统桥接
- 安全存储
- 原生通知
- 窗口/托盘/快捷键
- 自动更新

### 3. 暂时不要共享的模块

- 过度依赖 Next.js server/runtime 的逻辑
- 直接依赖浏览器环境的上传或 auth 包装
- 强耦合 web 路由结构的页面外壳

## 页面优先级建议

第一批页面建议只做这些：

1. 登录页
2. 主工作区文件列表
3. 上传入口
4. 转写详情页
5. 设置页

下面这些可以延后：

- 复杂计费页
- 营销或引导页
- 所有低频管理页

## 风险点

### 1. 当前 web 与 Next.js 运行时耦合较深

如果直接复用页面级代码，desktop 很容易被 Next 特有能力绑住。

应对：

- 先复用领域层和组件层
- 页面壳和路由单独在 `desktop/` 实现

### 2. 上传和鉴权链路在桌面端行为不同

浏览器里的 cookie、storage、页面刷新语义，在桌面 WebView 中不完全等价。

应对：

- 单独设计 session 持久化
- 先用最稳妥的 token/session 流程跑通 MVP

### 3. 原生能力容易让项目范围失控

桌面项目很容易被“顺手加一个托盘、快捷键、录音、离线同步”拖慢。

应对：

- MVP 只做对主路径直接增益的能力
- 所有原生特性必须绑定明确场景和验收标准

## 人力与节奏建议

如果是 1 到 2 名开发者，推荐节奏：

- 第 1 周：Phase 0
- 第 2 到 3 周：Phase 1
- 第 4 周：Phase 2
- 第 5 周：收敛问题、补测试、准备内测

如果只有 1 人兼职推进，建议先把目标压缩到：

- 登录
- 文件列表
- 上传
- 转写详情
- 系统通知

## 第一批任务清单

可以直接从下面这些任务开工：

1. 新建 `desktop/`，跑通 Tauri 2 + React + Vite
2. 设计 desktop 的目录结构和路由骨架
3. 从 `app/` 抽出 `shared-types`
4. 从 `app/src/services/api.ts` 抽出可复用的 `shared-api`
5. 打通 `me/files/file detail` 三个核心接口
6. 实现本地文件选择与上传
7. 做转写状态轮询与完成通知
8. 补一版内测发布流程

## 推荐结论

`plaud desktop` 最合理的做法，不是把现有 web 直接打包，而是：

- 保留 `app/` 作为 web + backend 主工程
- 新建 `desktop/` 作为独立桌面客户端
- 优先抽共享的类型、API 和业务模型
- 用 `Tauri 2` 承载文件系统、通知、更新和桌面交互

这样做短期成本略高于“包壳”，但中期维护性、性能和产品空间会更好。
